# Time Analytics 詳細設計（HPlus Analytics v1.1 Phase B-2）

## 0. 位置付けと基本思想

Time Analytics は「どの曜日の売上が高いか」をランキングする機能ではない。店長が、出勤量を考慮しても強い曜日、効率が高い曜日、出勤を増やす余地のある曜日を判断できるようにする。

評価順序は **Sample → Efficiency → Volume**。曜日ごとの対象日数や出勤回数が不足する場合はData不足とし、改善提案を強く出さない。原因分析の後に提案する `Cause before Action` を適用し、予約元や機会損失を推測しない。

## 1. 店長の質問

- 本当に強い曜日はどれか（総量と効率を分離）。
- 出勤人数を考慮しても強い曜日はどれか。
- 効率の高い曜日はどれか。
- 出勤を増やすならどの曜日か。
- 曜日を変えた方が伸びるCastは誰か。
- 現状の曜日差は母数・掲載・制約で説明できるか。

## 2. 対象範囲と単位

- 店舗：全体、春日部、越谷、野田（CTI補助）。
- 全体範囲：CTIは3店舗、Townは春日部＋越谷、Heavenは春日部。
- `mergedIntoCastId IS NULL` のCastのみ通常対象。
- 店舗・曜日・businessDate単位で自然キー重複を除き、同一Castの同日複数店舗出勤は全体出勤日数では1日とする。
- 対象外媒体は0にせず `—`。データ未取得日と実績0を区別する。

### 2.1 主なデータソース

- CTIの出勤・成果：`CtiCastDaily` と確定済みBatch。
- Town露出：`TownCastDaily`、必要に応じてURL／LPの日次値。
- Heaven露出・活動：`HeavenCastDaily` の `DAILY_EVENT`／`SNAPSHOT` とraw status。
- 属性・制約：`Cast`（startedOn/endedOn、ランク）、`MediaListing`、取得できる稼働・時間項目。
- `ImportError`、未確定BatchはData不足・警告の根拠として表示し、値を0に置換しない。

## 3. 曜日集計の共通ルール

### 3.1 曜日の定義

businessDateを店舗のタイムゾーンで曜日化する。曜日名、対象日数、最終対象日を表示する。ISO週など週次定義はTrend Analyticsと共通化する。

### 3.2 Volume（曜日の規模）

| 指標 | 定義 |
|---|---|
| 売上 | 対象曜日のCTI料金合計 |
| 女子報酬 | 対象曜日の女子報酬合計 |
| 利益 | 保存された利益値のみ。未保存なら— |
| 予約・接客・本指名 | CTI実績の合計 |
| 出勤人数 | 対象曜日に1回以上出勤したCastのユニーク人数 |
| 総出勤時間 | attendanceMinutes合計÷60 |
| Town PV/UU | 対象店舗・曜日のTown値。日別UUを再ユニーク化しない |
| Heavenアクセス | `DAILY_EVENT`の対象曜日合計 |
| 写メ日記 | 利用可能な確定値の合計 |

### 3.3 Efficiency（曜日の生産性）

| 指標 | 式 | 分母不足 |
|---|---|---|
| 売上/人 | 売上÷出勤人数 | 出勤人数0は— |
| 売上/時間 | 売上÷総出勤時間 | 時間0は— |
| 報酬/時間 | 女子報酬÷総出勤時間 | 時間0は— |
| 予約/人 | 予約÷出勤人数 | 人数0は— |
| 予約/時間 | 予約÷総出勤時間 | 時間0は— |
| 平均単価 | 売上÷接客数（正式定義に従う） | 接客0は— |
| 本指名率 | 本指名÷接客数 | 接客数が少ない場合Low |
| 稼働率 | 正式な稼働分母がある場合のみ | 推定禁止 |

### 3.4 Sample（曜日の信頼性）

- 対象曜日日数
- 延べ出勤人数
- ユニークCast数
- 総出勤時間
- CTI実績日数、Town／Heaven取得日数
- 比較対象曜日数
- Confidence

基本ConfidenceはHigh（20日以上）、Medium（10–19）、Low（5–9）、Insufficient（4日以下）。キャスト曜日分析では対象曜日の出勤回数を主Sampleとし、店舗曜日分析では対象曜日の日数を主Sampleとする。母数が異なる場合はラベルで明示する。

## 4. 店舗曜日分析

### 4.1 表示カード

曜日ごとにVolume、Efficiency、Sampleを並べ、総売上と売上/時間を別カードにする。「総量が最大」「効率が最大」「比較可能性が高い」を別ラベルで表示する。

### 4.2 ヒートマップ

- 行：指標、列：曜日。
- VolumeとEfficiencyはタブを分け、同じ色尺度で混ぜない。
- 色は順位ではなく、選択したBaseline（店舗中央値、平均、前月同曜日など）からの乖離を表す。
- Sample不足セルは色付けせずData不足とする。

### 4.3 曜日推移

期間内の日付を横軸とし、曜日色分けは補助情報に留める。対象日数が異なる曜日を単純比較せず、日次値と曜日集計を併記する。

## 5. キャスト曜日分析

### 5.1 Volume

出勤回数、出勤時間、売上、女子報酬、予約、接客、本指名を曜日別に表示する。同一日複数店舗はCastの出勤回数では1回とし、店舗別売上等は実績店舗単位で合算する。

### 5.2 Efficiency

売上/時間、報酬/時間、予約/時間、予約/出勤、平均単価、理論最大時給達成率を表示する。理論最大時給はランク・料金体系の基準値と実績時給を区別し、オプションを基準値に自動加算しない。

### 5.3 曜日適性

曜日適性は次のBaseline差を組み合わせる。

1. 本人の他曜日平均
2. 店舗の同曜日平均／中央値
3. 同ランクの同曜日平均
4. 近い出勤時間帯・出勤日数のCast

時給、予約/時間、Sampleを併記する。母数不足は適性を表示せずData不足とする。本人平均との差だけで曜日変更を断定しない。

## 6. 曜日改善提案

### 6.1 判定フロー

1. 対象曜日のSampleとConfidenceを確認。
2. Efficiencyを本人・店舗・同ランクBaselineと比較。
3. 既存出勤量、稼働率、理論最大時給達成率、Capacityを確認。
4. Volumeの増加余地を確認。
5. 原因 → 根拠 → 改善候補の順で、原則1つのNext Best Actionを提示。

### 6.2 例

「金曜日の時給が本人平均より+20%、対象回数24回（High）」は出勤増加候補の根拠になり得る。ただし既に金曜出勤が多い、稼働率が高い、Capacity上限が近い場合は、出勤増加を提案せずSchedule制約の確認を優先する。

「売上が高いが出勤人数・時間も多く、売上/時間が中央値以下」は、強い曜日とは断定せず、量による総額と効率を分離して表示する。

## 7. 時間帯分析の調査境界

### 7.1 現時点で算出可能な範囲

現行の設計で確実に扱えるのは、CTIに保存されたbusinessDate、出勤時間（attendanceMinutes）、実績値を使った日次・曜日・出勤時間帯の集約までである。時間帯別の予約・接客・売上を算出するには、実績に開始時刻または時間帯キーが保存されていることが必要である。

### 7.2 算出不可・推測禁止

開始・終了時刻、予約時刻、媒体閲覧時刻がない場合、午前／午後／深夜の成果や媒体経由予約を推測して表示しない。「時間帯分析不可：時刻粒度なし」と表示する。attendanceMinutesだけから時間帯別売上を按分しない。

### 7.3 将来拡張

時刻データが追加された場合も、曜日分析と同じSample → Efficiency → Volume、Baseline、Confidenceを適用し、曜日×時間帯のセルごとに対象件数を保持する。

## 8. 比較条件とUIカード

- 比較条件カード：期間、店舗、曜日、Baseline、主Sample、対象日数、確定Batch数。
- Confidenceカード：High/Medium/Low/Insufficientの理由。
- Data不足カード：欠損日、未確定Batch、媒体取得日数不足。
- 比較カード：本人平均との差、店舗平均との差、同ランク平均との差、前月同曜日。
- Insightカード：原因、根拠、制約、改善候補、詳細リンク。

## 9. Correlation Analyticsへの接続

将来の相関分析に備え、曜日別・Cast別に次の観測単位を保持する。

- metricKey、businessDate、weekday、storeId、castId、value、valueKind、rawValueStatus
- 曜日対象日数、出勤回数、総時間、媒体取得日数、Confidence
- Baseline識別子と比較期間

候補ペア（写メ日記→PV、PV→予約、出勤時間→売上、本指名率→売上など）は、同一期間・同一Cast／店舗の観測値として相関表示する。予約元や因果を復元しない。

## 10. Phase B-2実装境界と未確定事項

共通型候補は `TimeBucket`（day/week/month/weekday）、`BaselineDescriptor`、`SampleSummary`、`Confidence`、`TimeMetricPoint`、`WeekdayInsight`。集計は純粋関数、表示・AI Insight・データ取得を分離する。

未確定事項：業務週定義、祝日・休業日の扱い、稼働率分母、時間帯データの有無、前年同月の利用開始時期、利益の正式カラム。
