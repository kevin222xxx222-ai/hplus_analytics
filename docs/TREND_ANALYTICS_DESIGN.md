# Trend Analytics 詳細設計（HPlus Analytics v1.1 Phase B-2）

## 0. 位置付けと設計原則

Trend Analytics は「増えた／減った」を表示するだけの画面ではない。変化を Volume（量）・Efficiency（効率）・Sample（母数）の3軸に分解し、原因候補、制約、次に確認すべき行動を根拠付きで示す読み取り専用の分析である。

評価順序は必ず **Sample → Efficiency → Volume** とする。母数が不十分な場合は、差分が大きくても好調・不調を断定しない。改善提案は **Cause before Action**（原因分析の後に提案）に従い、媒体経由予約や機会損失など取得できない事実を推測しない。

共通定義は `docs/ANALYTICS_DESIGN_PRINCIPLES.md` を正とし、本書はTrend固有の期間比較と分解を定義する。対象は `mergedIntoCastId IS NULL` の通常Castおよび確定済み実績とする。

## 1. 店長の質問と回答単位

| 質問 | 分析単位 | 主な回答 |
|---|---|---|
| 前月より良くなった原因は？ | 店舗・全体・Cast・媒体 | Volume増かEfficiency改善か、Sampleの十分性 |
| 前週より悪くなった原因は？ | 同上 | 出勤量、時間、予約・成約、露出の寄与候補 |
| 改善は継続しているか？ | 日次・週次・月次 | 連続期間の方向、単発外れ値、信頼度 |
| 売上だけでなく効率も改善したか？ | 比較対象ごと | 売上差、出勤時間差、売上/時間差を同時表示 |

因果は断定せず、「観測された差」「整合する原因候補」「未確認の制約」を分離する。

## 2. 対象スコープと媒体範囲

- 店舗フィルタ：全体、春日部、越谷、野田（CTI補助）。初期値は全体。
- 全体のCTI：春日部・越谷・野田、Town：春日部・越谷、Heaven：春日部。
- 店舗別の対象外媒体は0ではなく `—` と表示し、範囲外を欠損扱いしない。
- CTIは確定済み実績を使用し、未確定・FAILED BatchはDATA HEALTH警告にする。
- Heavenは `DAILY_EVENT` を期間合計、`SNAPSHOT` を期首・期末・増減で扱う。
- 同一自然キーを重複加算しない。Cast統合元は通常集計から除外する。

### 2.1 分析ディメンション

同じ期間比較を次の粒度で再利用する。

- 全体：対象店舗範囲を合算。媒体範囲を注記。
- 店舗：storeId単位。主所属ではなく実績行のstoreIdを使用。
- Cast：castId単位。統合元は最終Castへ解決済みの場合のみ最終Cast側で表示。
- 媒体：CTI、Town、Heavenを別系列で表示。媒体間の予約経路は結合しない。
- ランク：Castの対象期間時点のランク属性。ランク欠損は比較不能とする。

### 2.2 主なデータソース

- CTI：`CtiCastDaily` および確定済み `ImportBatch` の範囲・status。
- Town：`TownCastDaily`、`TownUrlDaily`、`TownLandingDaily`（媒体・店舗範囲を分離）。
- Heaven：`HeavenShopDaily`、`HeavenCastDaily` と `valueKind`／`rawValueStatus`。
- 共通属性：`Cast`、`CastAlias`、`MediaListing`。Aliasの有効期間は対象日判定にのみ使用。
- 健全性：`ImportError` と未確定／FAILED Batch。分析値を黙って補完するためには使用しない。

## 3. 比較軸と期間の定義

### 3.1 比較軸

| 軸 | 定義 | 注意 |
|---|---|---|
| 前日比 | 同一集計単位の直前businessDateとの差 | 休業日・欠損日は比較不能と表示 |
| 前週比 | 当週累計と直前週の対応期間 | 集計日数を揃える |
| 前週同曜日比 | 同曜日の直前週との差 | 曜日要因を保つ |
| 前月比 | 完了した同じ期間単位の前月との差 | 月途中は使用しない |
| 前月同期間比 | 当月1日〜対象日と前月1日〜同経過日 | 月途中の標準比較 |
| 日別推移 | businessDate単位 | 欠損日は空白／データなし |
| 週別推移 | ISO週または業務週単位 | 採用カレンダーを表示 |
| 月別推移 | 暦月単位 | 期間日数を併記 |
| 前年同月比 | 同定義の前年同月 | 将来拡張、データ不足時は比較不能 |

### 3.2 比較期間の整合

比較対象は対象日数、店舗範囲、媒体範囲、確定状態、指標定義を揃える。揃えられない場合は比較値を作らず、理由（例：前月のHeaven取得日数不足）を表示する。単純な当月累計と前月全月を比較しない。

## 4. 画面・カード設計

### 4.1 共通ヘッダー

- 期間、店舗、対象媒体範囲、比較軸、データ健全性を表示。
- 比較対象の期間・対象日数・確定Batch数を明示。
- 「この画面で分かること」「次に見るページ」へのガイドを置く。

### 4.2 変化サマリー

各指標について次を1カードにまとめる。

- 現期間Volume、比較期間Volume、差額、差異率
- Efficiency（売上/出勤時間など）、比較期間との差
- Sample（対象日数、出勤人数、総出勤時間、取得日数、Confidence）
- 分解ラベル：`量の変化`、`効率の変化`、`母数不足`、`比較不能`
- 原因候補、根拠指標、注意事項、関連詳細リンク

### 4.3 Trend分解

売上などOutputの差を、次の順で説明する。

1. 出勤人数・出勤日数の差
2. 総出勤時間の差
3. 売上/出勤者、売上/時間の差
4. 予約・接客・成約・本指名など成果内訳
5. Town PV/UU、Heavenアクセス、日記など参考露出

これは寄与額の厳密な因果分解ではなく、同期間の観測比較であることを明示する。

### 4.4 日次・週次・月次グラフ

- 総量系列と効率系列を同一軸に重ねず、上下段または切替で表示。
- X軸は日付／週／月、Y軸は指標名と単位を表示。
- 欠損・未確定日は0に補完せず、空白とDATA HEALTHリンクを表示。
- グラフ下に「分かること」「見るポイント」「注意点」「関連ページ」を表示。

## 5. 指標設計

### 5.1 Volume

売上、女子報酬、利益（保存値がある場合）、予約、接客、成約、本指名、出勤人数、出勤日数、出勤時間、Town PV/UU/TEL、Heavenアクセス、写メ日記を対象とする。

### 5.2 Efficiency

売上/出勤者、売上/出勤日、売上/時間、報酬/時間、予約/出勤者、予約/時間、成約/時間、平均単価、本指名率、PV/出勤日、Heavenアクセス/出勤日を対象とする。分母0または媒体値不存在はnull（`—`）。

### 5.3 SampleとConfidence

分析種別に応じて母数を選ぶ。

| 分析 | 主Sample | 補助Sample |
|---|---|---|
| 店舗日次 | 対象日数 | 出勤人数、総時間 |
| 店舗週次・月次 | 対象期間数 | 取得日数 |
| Cast推移 | 出勤回数または実績日数 | 総出勤時間 |
| 媒体比較 | 両期間で値が存在する日数 | Cast数 |

基本ConfidenceはHigh（20以上）、Medium（10–19）、Low（5–9）、Insufficient（4以下）。Insufficientは強い判定・施策優先順位に使用しない。

## 6. 変化原因の判定フロー

1. Sampleと確定状態を確認。
2. 比較期間・範囲・基準値を固定。
3. Efficiencyを比較し、効率改善／悪化を判定。
4. Volumeを比較し、人数・時間・成果のどこが動いたかを見る。
5. 制約（稼働率、Capacity、出勤可能時間）を確認。
6. 原因候補と信頼度を表示し、Next Best Actionは原則1件だけ提示。

「売上増＝効率改善」「アクセス増＝予約増」とは解釈しない。媒体別予約元、断った予約、機会損失は原因・効果額として扱わない。

## 7. AI Insight出力

表示順は **原因 → 根拠 → 改善候補**。各Insightに比較対象、差分、Sample、Confidence、未確認の制約を添える。

例：

> 原因候補：売上増は出勤時間増の影響が大きい。根拠：総時間は+18%、売上/時間は+1%。Sample：対象18日、Confidence Medium。改善候補：効率維持を確認しながら時間拡大を検討。

断定語（必ず、原因は、効果がある）は避け、「可能性」「整合する」「参考」とする。

## 8. 将来のCorrelation Analytics接続

Trendの時系列出力は、将来の相関分析へ渡せるよう次の共通単位を保持する。

- 指標キー、businessDate、storeId、castId、value、valueKind、rawValueStatus
- 比較期間ID、対象日数、欠損日数、Confidence、対象範囲
- `diary_posts → PV`、`PV → 予約`、`出勤時間 → 売上` 等の候補ペアは、同一日・同一Cast／店舗の観測値として保存し、因果ではなく相関として表示する。

相関の実装・有意性判定・施策効果推定は本フェーズの対象外。

## 9. Phase B-2の実装境界

将来は `TrendMetricPoint`（純粋な値）、`ComparisonWindow`（期間）、`BaselineDescriptor`（比較対象）、`SampleSummary`（母数）、`Confidence`、`InsightEvidence`、`NextBestAction` を共通型として定義する。集計関数は副作用のない純粋関数とし、確定・取込・Aliasサービスから分離する。
