# HPlus Analytics 共通分析設計原則

## HPlus Analytics Philosophy

HPlusは売上を可視化するだけのシステムではない。売上という結果だけでなく、原因、制約、改善余地を分析し、店長が次に取るべき行動を根拠付きで提示する分析システムである。

すべての分析は、まずSampleを確認し、次にEfficiencyを確認し、最後にVolumeを確認する。母数が不十分な場合は強い結論を出さず、原因を確認してから改善提案へ進む。

## 1. 目的と適用範囲

本書は、HPlus Analytics v1.1以降の分析に共通して適用する設計原則である。Performance Funnel、Trend Analytics、Time Analytics、Store Analytics、Cast Analytics、AI Action Engine、予測機能は、特別な理由がない限り本書の定義を使用する。

分析の目的は、数字の大小をランキングすることではなく、量・効率・母数・制約を同時に確認し、次に確認すべき事実または施策候補を明確にすることである。

### 1.1 分析の前提

- 分析は読み取り専用の派生値であり、既存の取込値・実績値・KPI計算を変更しない。
- `mergedIntoCastId IS NULL` を通常のキャスト集計対象とする。
- 媒体に値が存在しない場合は0ではなくnull（画面表示は「—」）とする。
- 分母が0、または必要な分母が欠損する率・効率はnullとする。
- 未確定Batch、FAILED、OPENエラーがある場合はデータ健全性警告を表示するが、分析処理を黙って停止しない。
- Town・Heavenの露出とCTIの予約・成約は顧客単位で対応しない。媒体経由予約・媒体経由成約を作らない。

## 2. Analysis Priority（分析優先順位）

分析軸はVolume、Efficiency、Sampleの3つだが、評価順序は次のとおりとする。

1. **Sample**：この分析は十分な母数を持つか
2. **Efficiency**：同じ条件あたりの成果はどうか
3. **Volume**：総量・規模はどうか

Sample不足の分析は、効率や売上が高くても「好調」「改善必須」などの強い結論を出さない。ConfidenceはSample判定の結果であり、EfficiencyやVolumeの値を後から正当化するものではない。Insufficientの場合は値を表示しても、候補判定・施策優先順位・AIの強い提案には使わない。

## 3. 3軸評価：Volume / Efficiency / Sample

すべての判定は、原則として次の3軸を同時に評価する。絶対値だけで「良い」「悪い」を判定しない。

### 3.1 Volume（量）

候補指標：

- 売上、女子報酬、CTI利益
- 予約数、接客数、成約数、本指名数、写真指名数、フリー数
- 出勤人数、出勤日数、出勤時間
- Town PV、Town UU、Town TEL
- Heaven PAGE_ACCESS、MY_GIRL、写メ日記数

Volumeは規模・総量を示す。売上が大きくても出勤人数や時間が多いだけの場合があるため、必ずEfficiencyと併記する。

### 3.2 Efficiency（効率）

基本指標：

- 売上／出勤者、売上／出勤日、売上／出勤時間
- 女子報酬／出勤日、女子報酬／出勤時間
- 予約／出勤者、予約／出勤日、予約／出勤時間
- 接客／出勤日、接客／出勤時間
- 本指名率、成約率、平均単価
- PV／出勤日、PV／出勤時間
- Heavenアクセス／出勤日、Heavenアクセス／出勤時間
- 写メ日記1件あたりPV、写メ日記1件あたり成果（成果は定義済みのCTI値のみ）

率・効率は必ず分母とSampleを表示する。Town予約率、Heaven予約率、Town経由予約、Heaven経由予約は定義しない。

### 3.3 Sample（母数・信頼性）

各分析結果に、最低限次を表示する。

- 対象日数
- 出勤回数または出勤日数
- 出勤人数
- 総出勤時間
- 媒体データ取得日数
- キャスト数
- 曜日別対象回数
- 比較対象件数

同じ「20件」でも、曜日分析の日数とキャスト比較人数では意味が異なるため、サンプル種別を明記する。

### 3.4 曜日分析の原則

「売上が高い曜日」と「効率が高い曜日」は同じ意味ではない。曜日の優劣を絶対売上だけで判定しない。

曜日分析では、次を同時に表示する。

- Volume：総売上、予約数、出勤人数、総出勤時間
- Efficiency：売上／出勤者、売上／出勤時間、予約／出勤者、予約／出勤時間
- Sample：対象日数、総出勤時間、対象キャスト数、曜日別対象回数、Confidence

出勤人数や時間が多い曜日は総売上が高くなり得るため、VolumeとEfficiencyを分離する。対象日数が少ない曜日は、Confidenceを下げて断定的な施策提案を避ける。

## 4. KPI分類

AI Action Engine、Growth Potential、予測機能は、KPIを役割別に扱う。

### Input KPI

店長が運用・掲載・シフトで直接改善し得る入力側指標。

- 出勤日数、出勤時間
- Town PV、Town UU、Heavenアクセス
- 写メ日記数
- 掲載媒体数、掲載状態

### Output KPI

入力の結果として観測する成果指標。

- 売上、女子報酬、利益
- 予約、接客、成約、本指名

### Efficiency KPI

InputとOutputの関係を同じ母数で比較する指標。

- 売上／時間、売上／人
- 女子報酬／時間
- 予約／時間、予約／人
- 接客／日、成約／日
- 本指名率、成約率、平均単価

### Constraint KPI

改善の上限・制約を示す指標。

- 稼働率、出勤可能時間
- 現在時給、基準理論最大時給、理論最大時給達成率
- Schedule制約、Capacity制約

Inputを変えればOutputが必ず変わるとは扱わない。Constraintを先に確認し、KPIの役割を混同しない。

## 5. Confidence Rules

### 5.1 基本ランク

| 信頼度 | 基本サンプル数 | 表示・利用方針 |
|---|---:|---|
| High | 20以上 | 通常の参考候補として表示 |
| Medium | 10〜19 | 参考候補。注意書きを表示 |
| Low | 5〜9 | 強い施策提案を避け、確認候補として表示 |
| Insufficient | 4以下 | 判定・ランキングから除外し「データ不足」 |

### 5.2 指標別の母数

基本ランクをそのまま適用せず、分析種別ごとに母数を選ぶ。

- 曜日別キャスト分析：対象曜日の出勤回数
- 店舗曜日分析：対象曜日の日数
- 媒体相関：両方の値が存在する日数
- キャスト比較：比較対象キャスト数
- 本指名率：接客数（標準5件未満はLow/除外）
- 成約率：予約数（標準5件未満はLow/除外）
- 時間効率：総出勤時間（0は除外）
- PV／日：Town PVが存在する出勤日数

最低母数を満たさない場合、値そのものは表示しても「判定根拠」には使わない。

## 6. Baseline（比較基準）

分析結果には必ず「何と比較したか」を表示する。

### 6.1 キャスト分析

優先順位は次のとおりとする。

1. 本人の同期間・同曜日または本人過去平均（十分なSampleがある場合）
2. 同店舗・同期間の中央値
3. 同店舗・同期間の平均
4. 同ランク平均
5. 出勤日数・出勤時間が近いキャストの平均

本人比較はSample不足なら店舗基準へフォールバックし、フォールバック理由を表示する。

### 6.2 店舗分析

優先順位は次のとおりとする。

1. 前週同曜日
2. 前月同期間
3. 前月全体
4. 過去同曜日平均
5. 他店舗平均・全店舗平均

単純な当月累計と前月全月の比較は、月途中では使用しない。

## 7. Comparison Rules（比較軸）

| 比較軸 | 定義 |
|---|---|
| 前日比 | 当日値 ÷ 前日値 − 1 |
| 前週比 | 当週累計または同じ集計単位を前週対応期間と比較 |
| 前週同曜日比 | 同じ曜日の直前週との比較 |
| 前月比 | 完了月同士など、同じ期間単位の比較 |
| 前月同期間比 | 当月1日〜対象日と前月1日〜同経過日を比較 |
| 月次推移 | 月単位の同一指標の推移 |
| 週次推移 | ISO週または業務週単位の推移 |
| 日次推移 | businessDate/date単位の推移 |
| 曜日別平均 | 同じ曜日の値の平均。対象日数を併記 |
| 前年同月比 | 将来拡張。前年同月の同定義データと比較 |

月途中の前月比較では、前月全月を分母にしない。対象期間、経過日数、営業日数を画面上で確認できるようにする。

## 8. 制約条件の扱い

改善候補は、次の条件を踏まえて順位付けする。

- 出勤日数、出勤時間
- 稼働率または接客負荷（正式値がある場合のみ）
- 現在時給、基準理論最大時給、達成率
- 予約数、接客数
- 媒体露出、写メ日記数
- Sample数、データ健全性

稼働率・接客時間・最大枠が保存されていない場合、稼働率を推定して断定しない。「高負荷の可能性」「制約を確認」と表現する。

売上・アクセスが高いキャストへ常に露出施策を提案しない。時給達成率が高く、出勤時間が短い場合は、露出より出勤時間・出勤日数の延長を優先候補とする。

## 9. Cause before Action

AI Action EngineとGrowth Potentialは、改善案を先に選ばず、原因仮説を先に評価する。

1. Sampleとデータ健全性を確認
2. Input・Output・Efficiency・Constraint KPIを分離
3. 観測された制約と比較基準との差を特定
4. 原因候補の信頼度を評価
5. その後にNext Best Actionを1件選択

例として、稼働率98%、理論最大時給達成率が高く、出勤時間が短い場合は、露出不足とせず「Schedule制約の可能性」として出勤時間延長を優先する。

現在取得できない情報は原因として扱わない。

- Town経由予約
- Heaven経由予約
- 断った予約
- 機会損失額

これらを推定値で補完して、原因・損失・施策効果を断定してはならない。

## 10. 理論最大時給

### 10.1 定義

理論最大時給は、ランク・基本料金・基本女子報酬から算出する基準上限であり、予測値や保証値ではない。

業務上の初期目安：

- プラチナ：90分売上23,000円、基本女子報酬11,500円、約7,600円/時
- レギュラー：90分売上20,000円、基本女子報酬9,500円、約6,300円/時。運用表示では約6,000円前後の目安と注記

オプションはフルバックでも変動要素のため、基準理論最大時給には原則含めない。

### 10.2 表示する値

- 基準理論最大時給：ランク・基本料金体系からの値
- OP込み実績時給：女子報酬と対象OPを分離取得できる場合のみ
- 現在実績時給：女子報酬 ÷ 出勤時間
- 理論最大時給達成率：現在実績時給 ÷ 基準理論最大時給

ランク・料金マスタ未設定、出勤時間0、女子報酬欠損はnullとする。料金体系をアプリコードへ固定せず、将来はランク・料金マスタから取得するサービス境界にする。

## 11. Growth Potential

Growth Potentialは、売上が低い順ではなく、制約と改善余地の組み合わせで判定する。

### 11.1 分類

- **Exposure不足**：CTIアクティブ、媒体データあり、媒体露出効率が基準未満。未掲載は「掲載状態確認」と分離。
- **Activity不足**：出勤日数・時間が少ないが、効率指標に一定の根拠がある。
- **Efficiency改善余地**：露出・活動が一定量あり、複数の成果効率が基準未満。
- **Schedule制約**：現在時給・達成率が高く、出勤時間が少ない／延長余地の確認が必要。
- **Capacity上限**：正式な稼働率・枠消化情報が高い場合のみ。正式値がなければ候補化しない。
- **Data不足**：Sample不足、媒体欠損、未確定Batch、Alias未解決。
- **安定維持**：Volume・Efficiency・Sampleが基準以上で、急な改善課題がない。

### 11.2 判定優先順位

複数条件が同時に成立した場合は、次の順序で上位分類を最終結果として採用する。

1. **Data不足**
2. **Capacity上限**
3. **Schedule制約**
4. **Exposure不足**
5. **Activity不足**
6. **Efficiency改善余地**
7. **安定維持**

この順序はAnalytics Engineの`GROWTH_PRIORITY`と一致させる。Data不足では他分類を強く判定しない。Capacity上限では露出・活動量の追加を優先せず、Schedule制約では実行可能な勤務条件を先に確認する。Exposure不足、Activity不足、Efficiency改善余地は、必要なSampleと比較可能な入力がある場合に限り採用する。重大な不足・制約がない場合のみ安定維持とする。

データ不足を強い施策に変換しない。複数分類に該当した場合は最優先分類1つと、根拠指標を表示する。

## 12. Next Best Action

改善提案は原則1キャスト1件だけ表示する。候補を羅列せず、優先順位ルールで最重要の一手を選ぶ。

出力必須項目：

- 提案内容
- 判定理由
- 使用指標と値
- 比較対象・比較値
- Sample種別と件数
- 信頼度
- 注意事項・因果関係を示さない注記

優先順位の例：

Growth Potentialの上記優先順位を、Next Best Actionの原因選定にもそのまま適用する。Insufficientは提案なし、Low ConfidenceはREFERENCE（参考）扱い、安定維持は原則として提案なしとする。いずれも1キャストにつき最重要1件だけを返す。

Cause before Actionに従い、分類（原因）→根拠指標→比較対象・Sample→Actionの順で構成する。Capacity上限・Schedule制約では露出増加を提案しない。

取得できない情報は算出・推定しない。

- Town経由予約数
- Heaven経由予約数
- 断った予約件数
- 機会損失額

## 13. 禁止する誤分析

- 絶対売上だけで良し悪しを決める
- Sampleを表示せず率・効率を比較する
- 分母0を0として扱う
- 媒体未掲載を0として下位判定する
- Town予約率、Heaven予約率、媒体経由成約を作る
- Town PV・Heavenアクセスを顧客単位で合算して正式ファネルと呼ぶ
- 月途中の当月累計と前月全月を比較する
- 1日・少数件の値をHigh Confidenceとして扱う
- 現在時給・稼働率・理論最大時給を混同する
- AI文章で算出値、根拠、制約を上書きする
- 予測値や改善候補を事実・保証として表示する

## 14. Phase B-2以降の共通実装境界

### 14.1 型

```text
AnalysisDimension = VOLUME | EFFICIENCY | SAMPLE | BASELINE | CONSTRAINT | GROWTH
Confidence = HIGH | MEDIUM | LOW | INSUFFICIENT
ComparisonBasis = SELF_PERIOD | PREVIOUS_PERIOD | STORE_MEDIAN | STORE_MEAN | TOP25 | PEER_GROUP
MetricValue = { value: number | null, unit, numerator, denominator, sample }
Evidence = { metricKey, value, baseline, gap, sample, confidence }
NextBestAction = { code, label, evidence[], priority, confidence, disclaimer }
```

### 14.2 純粋関数

- `calculateMetric(numerator, denominator)`：0除算・nullを統一
- `calculateSample(kind, records)`：分析種別ごとの母数を算出
- `classifyConfidence(sample, rule)`：High/Medium/Low/Insufficient
- `selectBaseline(scope, period, basis)`：比較対象とフォールバック理由を返す
- `comparePeriods(current, baseline, axis)`：前日・前週同曜日・前月同期間を統一
- `calculateTheoreticalHourlyRate(rank, priceRule)`：料金マスタを受け取る純粋関数
- `classifyGrowthPotential(metrics, constraints, confidence)`：分類と根拠を返す
- `selectNextBestAction(evidence)`：1件の施策コードを返す

### 14.3 サービス境界

- `analysis-data-service`：CTI・Town・Heaven・掲載・取込状態を読み取り、媒体別の正規化入力を返す
- `comparison-service`：期間、曜日、店舗、本人、PeerのBaselineを生成
- `confidence-service`：母数種別と信頼度を評価
- `constraint-service`：出勤、時間、稼働、時給、理論上限を評価
- `growth-potential-service`：Growth分類とEvidenceを生成
- `next-best-action-service`：定型施策コードを1件選択
- `insight-renderer`：算出値を文章化するが、値・根拠・免責は変更不可

これらは既存の取込・確定・Alias解決サービスとは分離し、分析時に書き込み処理を呼び出さない。
