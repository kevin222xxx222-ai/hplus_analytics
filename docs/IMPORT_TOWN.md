# デリヘルタウンCSV取込

更新日: 2026-07-21

## ローカルフォルダ一括取込

開発・ローカル環境ではADMINが`/imports/town/bulk`を利用できます。読み取り元は`TOWN_BULK_KASUKABE_DIR`と`TOWN_BULK_KOSHIGAYA_DIR`だけで、画面から任意パスは指定できません。Docker Composeでは両フォルダを同一絶対パスへread-only bind mountします。

ファイル種別は`dto.jp-shop-`=店舗、`dto.jp-gal-`=女子、`dto.jp-url-`=URL、`dto.jp-lp-`=LPです。店舗はファイル名ではなくフォルダで決定し、`(1)`は店舗判定に使用しません。`YYYYMMDD_to_YYYYMMDD`を対象期間とし、女子/URL/LPの複数日は取込不可です。

処理はクライアントから内部APIへ1ファイルずつ順次要求します。日付昇順、同日は店舗→女子→URL→LP順です。1件失敗しても後続を継続し、リロード後はImportBatchの状態・件数を再表示します。検証・確定には既存の`createTownPreview`、Town parser/resolver、preview JSON、`confirmTownImport`を使用します。元CSVは既存`UPLOAD_DIR`へコピーされ、ローカル元パスをImportBatchの`storagePath`には保存しません。

自動確定は構造、期間、種別、外部店舗IDが正常で、ERROR 0、女子/URL/LPを含む未紐付け0、曖昧0、完了済み同一SHAなし、同日別SHAなしの場合だけ可能です。未紐付けの除外、新規Cast作成、修正版候補の上書きは自動実行しません。

完了済み同一SHAは`SKIPPED_DUPLICATE`表示、未完了同一SHAは既存バッチへのリンク、同日・店舗・種別の別SHAは修正版候補警告とします。シンボリックリンク、通常ファイル以外、許可フォルダ外へ解決されるパス、CSV以外は解析しません。元ファイルは変更・削除しません。

CTI投入やAlias・在籍期間変更後は、未確定のTown一括バッチについて保存済み元CSVを再読込し、現在のCast、TOWN Alias、在籍期間でparser/resolverを再実行できます。同じbatch IDのpreview JSON、ImportError、ImportBatch検証件数だけを更新し、完了済みSHA重複とTown実績テーブルは変更しません。一括確定は別操作です。

Town詳細画面の元ファイル横に「このファイルを再解析」を表示します。FAILED / PREVIEW_READY / WAITING_FOR_CAST_LINKが対象で、STORE / CAST / URL / LANDINGの全種別に対応します。Bulk画面の「未確定Townバッチを全再解析」は対象バッチを1件ずつ順次呼び、1件失敗しても後続を継続します。単件・一括とも120秒タイムアウト、クライアント二重送信ロック、同一batchIdのサーバー実行集約を使用します。

### CTIを正とした未紐付け一括候補

`/imports/town/bulk`の「自動紐付け候補を解析」は、OPENの`UNMATCHED_CAST`を持つCAST / URL / LANDING行を店舗＋正規化名で横断集約します。解析は読み取り専用で、次の3区分を必ずプレビューします。

- A: 同一店舗のCTI Aliasまたは内部表示名と完全一致し、対象期間中の有効Castが1名だけ、Town Alias衝突・修正版・ID形式・曖昧がない候補
- B: 接頭辞「久」の有無、または`CastNameHistory`の過去名だけが異なり、同一店舗のCTI根拠から候補が1名に限定される候補
- C: ID形式、候補0/複数、期間外・無効Cast、別CastのTown Alias、修正版、人物名不明など。画面から一括実行不可

Aは全件、Bは管理者がチェックした候補だけを実行できます。実行前に候補全体のフィンガープリントを照合し、PostgreSQL advisory lockとSerializableトランザクション内で再解析します。条件が変わった候補、候補複数、別Cast Aliasは全処理を停止します。

実行は既存のTown後追い解決と同じ一意キー・保存処理を使用します。Town Aliasを作成または開始日前倒しし、同店舗・同一正規化名のpreview行とUNMATCHED_CASTを横断解決します。未確定バッチではTown実績を変更しません。確定済みバッチでも既存実績は上書きせず、まだ存在しない実績だけを追加します。件数は`metadata.insertedKeys`から累計再計算し、`metadata.importEvents`へ`BULK_CTI_TOWN_LINK`、区分、解決行数、追加件数、実行者、日時を記録します。

2026-07-21の全期間CTI確定後再解析では、未確定377バッチと新規候補1件を検証しました。自動確定可能99件、WAITING_FOR_CAST_LINK 96件、未紐付け4,467行、未紐付け正規化名84、店舗＋正規化名100、ERROR 0、完了済みSHA重複6件、同日別SHA修正版候補2件でした。一括確定は実行していません。

2026-07-21のCTI根拠一括候補の初回読み取り専用解析では、A 0人分/0行/0バッチ、B 2人分/124行/53バッチ、C 76人分/2,344行/274バッチでした。Bは春日部の「ゆあら→久ゆあら」87行/49バッチと「うゆ→久うゆ」37行/28バッチです。ID形式は43人分/918行/127バッチ、候補複数0、在籍期間外0、修正版候補2人分/316行/205バッチでした。紐付け実行は行っていません。

## 対象と操作

ADMINが `/imports/town` で店舗（春日部/越谷）、種別（店舗/女子/URL/LP）、対象期間、CSVを明示してアップロードします。ファイル名の `(1)` を含む文字列は店舗判定に一切使いません。選択した `ImportSource.store_id` が保存先の正で、URL中の外部店舗IDは矛盾検証だけに使います。

処理は、非公開保存 → 文字コード/構造/期間/数値検証 → プレビュー → 未紐付け解決 → 確定upsertです。元CSVとプレビューJSONは `UPLOAD_DIR` 配下に保存し、ADMINだけが取得できます。

## 対応CSV

| 選択種別 | ImportDataType | ヘッダー行 | 保存先 |
| --- | --- | ---: | --- |
| 店舗別 | `TOWN_STORE` | 3 | `town_store_daily` |
| 女子別 | `TOWN_CAST` | 4 | `town_cast_daily` |
| URL別 | `TOWN_URL` | 4 | `town_url_daily` |
| LP別 | `TOWN_LANDING` | 4 | `town_landing_daily` |

実ファイルはBOMなしCP932、カンマ区切り、LF改行でした。パーサーはUTF-8、UTF-8 BOM、CP932を安全に判定します。列順は固定せず列名で解決します。先頭行の `YYYY年M月D日 ～ YYYY年M月D日` を期間として検証します。

女子/URL/LPには行別日付がないため、対象開始日と終了日は同日必須です。複数日の値を日別へ推測配賦しません。店舗CSVは `日付` 列を使います。

## 検証とプレビュー

- 4種の既知ヘッダーを先頭50行から検出し、選択種別と異なれば取込不可
- 必須列欠損、対象期間不一致、期間外日付、負数、変換不能値はERROR
- 未知列、URL解析不能、比率の丸め幅超過はWARNING
- 変換不能値を0として保存しない
- URL解析不能は行を止めず `OTHER` として扱う
- `PV/UU` と `TEL/UU` は分母0なら `null`

プレビューには件数、取込可能/未紐付け、PV/UU/TEL合計、警告/エラー、行明細、URL/LPのページ種別を表示します。

## キャスト紐付け

対象店舗・対象日に限定し、次の順で解決します。

1. TOWN Alias原文一致
2. TOWN Alias正規化名一致
3. 在籍期間内の `Cast.normalized_name` 一意一致
4. 未紐付け/同名曖昧として保留

管理者は既存キャストへの紐付け（店舗限定TOWN Alias作成）、Town女子別の未紐付け行からの新規キャスト作成、今回除外、保留を選べます。URL/LP画面からは新規キャストを作成できません。Alias追加時は、同店舗・同一正規化名の未紐付け行へ適用します。女子CSVの確定行は `MediaListing(TOWN, store)` を掲載中へupsertしますが、CSVにないことを理由に非掲載へしません。

新規作成フォームは原文名、アップロード対象店舗、対象日を初期値とし、主所属店舗は春日部/越谷/野田/久喜/未設定、メモは任意です。久喜は所属先としてのみ選択でき、Town取込店舗にはなりません。作成時はCast、TOWN Alias、対象行、ImportError、同名の女子/URL/LP、未反映実績、バッチ件数、MediaListingを一操作で更新します。主所属が久喜でもAliasとMediaListingの店舗はアップロード対象店舗を使用します。

同じ正規化名で対象日に在籍中のキャストを事前検査し、1件なら既存キャストへの紐付けを推奨します。別人として作成する場合は確認ダイアログを必須とし、複数候補がある場合は新規作成を停止します。サーバーでもトランザクション内ロック後に同じ検査を再実行するため、画面判定だけで新規作成しません。

### 完了後の後追い解決

部分取込後もバッチは `COMPLETED_WITH_WARNINGS` のままです。確定済み行を未確定へ戻さず、OPENの `UNMATCHED_CAST` を持つ未紐付け行だけに解決UIを表示します。`COMPLETED`でも、旧データ等でOPEN未紐付けが残る場合は同じUIを表示します。

Alias追加時は、TOWN・対象店舗のAliasをupsertし、同じ店舗・同じ正規化名・Alias有効期間内のTOWN_CAST/TOWN_URL/TOWN_LANDINGプレビューを再解決します。別店舗へは適用しません。完了済みバッチでは新たに解決した行だけを各実績テーブルへ即時upsertします。URL/LPはアクセス値が既に保存済みの場合、同じ一意キーの行へ `cast_id` を反映します。

今回除外は対象バッチの対象行だけをSKIPPEDにし、対応エラーをIGNOREDにします。MediaListing、在籍状態、他バッチには影響しません。保留はOPENエラーとpendingに残します。

## 確定・再取込

エラーなしの解決済み行をトランザクションでupsertします。未紐付け行が残る場合も解決済み行だけの部分取込が可能で、バッチは警告付き完了になります。日付・店舗・各粒度の一意キーで最新値へ更新し、削除同期はしません。

同一ファイルハッシュの完了済みバッチがある場合は通常拒否し、管理者が明示した再処理だけ許可します。登録/更新/除外/未紐付け/警告/エラー件数と更新差分を `ImportBatch` に保持します。

後追い解決時の件数は単純加算しません。`metadata.insertedKeys / updatedKeys` と実績テーブルの一意キーを使い、現在状態の累計値として再集計します。操作履歴は `metadata.importEvents` に `INITIAL_CONFIRM`、`POST_RESOLUTION_IMPORT`、`POST_RESOLUTION_SKIP` として保持します。未紐付けとOPEN警告・エラーがなくなった時点でバッチを `COMPLETED` に更新します。

## 集計上の注意

正式値は `平均PV = ΣPV / ΣUU`、`TEL率 = ΣTELタップUU / ΣUU` です。店舗横断でも比率を単純平均しません。直帰率はUU加重です。Town TELとCTI予約・成約は個別顧客単位で直接対応しないため、画面では同期間の傾向比較として表示します。

実運用CSVは機密データとしてリポジトリへ格納しません。テストは匿名化した `tests/fixtures/town-*.csv` と、ファイルがローカルに存在する場合だけ実行する統合テストを使います。
